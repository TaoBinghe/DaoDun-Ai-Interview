## 条目 01：JVM、JDK、JRE 三者关系

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：运行环境与工具链
- 关键词：JVM、JDK、JRE、字节码、编译、运行时
- 难度：1

### 标准答案要点

1. `JVM` 是 Java 虚拟机，负责执行字节码，并提供类加载、内存管理、垃圾回收和安全机制。
2. `JRE` 是 Java 运行时环境，包含 JVM 和运行 Java 程序所需的核心类库。
3. `JDK` 是 Java 开发工具包，包含 JRE，以及 `javac`、`jdb`、`javadoc` 等开发工具。
4. 三者关系可以概括为：`JDK ⊃ JRE ⊃ JVM`。
5. Java 的跨平台本质上依赖于不同平台上都有对应实现的 JVM。

### 优秀回答范例

> JVM 是执行 Java 字节码的虚拟机，负责把 `.class` 文件加载并运行起来；JRE 是运行 Java 程序的最小环境，里面包含 JVM 和基础类库；JDK 则是在 JRE 基础上再加上编译器和调试器等开发工具，所以开发时装 JDK，运行时只需要 JRE。三者的包含关系可以理解成 JDK 包含 JRE，JRE 包含 JVM。

### 常见追问

- Java 为什么可以“一次编译，到处运行”？
- `javac` 编译后的产物是什么？
- JDK 8 与 JDK 11 以后在 JRE 分发方式上有什么变化？

### 评分要点

- 能否准确说清三者定义与包含关系。
- 能否说明跨平台能力主要由 JVM 提供。
- 表达是否有清晰结论，而不是只背概念。

### 易错点

- 把 JRE 说成包含编译器。
- 把跨平台完全归因于 Java 语言本身，而忽略 JVM 层。

## 条目 02：JVM 和 Java 有什么区别

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：语言与运行平台
- 关键词：Java语言、JVM、字节码、跨平台
- 难度：1

### 标准答案要点

1. Java 是一门编程语言及其生态规范。
2. JVM 是运行 Java 字节码的虚拟机，是运行平台的一部分。
3. Java 源码先编译成字节码，再由 JVM 在不同系统上执行。
4. Java 负责“怎么写”，JVM 负责“怎么跑”。

### 优秀回答范例

> Java 是开发者写业务代码用的编程语言，而 JVM 是运行 Java 字节码的虚拟机。Java 代码先通过编译器编译成 `.class` 字节码，再交给不同平台上的 JVM 去执行，所以 Java 的跨平台能力核心不是源码直接跨平台，而是字节码加 JVM 这一套运行机制实现的。

### 常见追问

- JVM 只能运行 Java 吗？
- Kotlin、Scala 为什么也能跑在 JVM 上？

### 评分要点

- 是否能区分“语言”和“运行平台”。
- 是否能说明编译为字节码再执行的流程。

### 易错点

- 把 Java 和 JVM 当成同一概念。
- 只说“一个写代码，一个跑代码”，但说不清字节码关系。

## 条目 03：Python 和 Java 有什么区别

- 岗位：Java后端开发
- 一级考点：语言对比
- 二级考点：编译执行模型、类型系统、应用场景
- 关键词：Python、Java、解释执行、字节码、静态类型、动态类型
- 难度：1

### 标准答案要点

1. Java 一般是静态类型语言，Python 是动态类型语言。
2. Java 源码通常先编译为字节码，运行在 JVM 上；Python 以解释执行为主，也会有字节码阶段，但运行模型不同。
3. Java 在大型企业级系统、并发和工程化上优势明显；Python 在脚本、数据分析、AI 生态上更强。
4. Java 对类型约束更强，适合中大型团队协作；Python 开发效率高，但运行性能通常弱于 Java。

### 优秀回答范例

> 我会从类型系统、运行方式和生态场景三个角度看。Java 是静态类型语言，编译期检查更强，代码先编译成字节码，再交给 JVM 执行；Python 是动态类型语言，开发效率高，语法更灵活。工程上，Java 更适合企业级后端、高并发服务，Python 更适合脚本、数据分析和 AI 相关场景。

### 常见追问

- 动态类型和静态类型在团队协作里各有什么优缺点？
- 你在后端场景下会怎么选 Java 或 Python？

### 评分要点

- 是否从多个维度比较，而不是只说“一个编译一个解释”。
- 是否结合实际应用场景说明差异。

### 易错点

- 只会背语法层面对比。
- 把 Python 简化成“完全不编译”的说法。

## 条目 04：Java 里是值传递还是引用传递

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：参数传递机制
- 关键词：值传递、引用副本、对象、方法参数
- 难度：2

### 标准答案要点

1. Java 只有值传递，没有真正意义上的引用传递。
2. 基本类型传递的是值的副本。
3. 对象类型传递的是“引用的副本”，不是对象本身。
4. 方法内部可以通过这个引用副本修改对象状态，但不能改变外部变量本身指向的新对象结果。

### 优秀回答范例

> Java 只有值传递。对于基本类型，传的是数值本身的副本；对于对象，传的是对象引用的副本。也就是说，方法里可以通过这个引用去改对象内部属性，但如果在方法里把参数重新指向一个新对象，外部引用并不会跟着变，这也是为什么很多人误以为 Java 有引用传递。

### 常见追问

- 为什么在方法里修改对象属性，外部能看到变化？
- 写一个交换两个 `Integer` 的方法能成功吗？为什么？

### 评分要点

- 是否说清“引用的副本”这几个字。
- 是否能区分“改对象内容”和“改引用指向”。

### 易错点

- 直接回答“对象是引用传递”。
- 不能解释为什么对象字段修改会生效。

## 条目 05：Java 的数据类型转换方式有哪些

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：类型转换
- 关键词：隐式转换、强制转换、装箱拆箱、parseInt
- 难度：1

### 标准答案要点

1. 自动类型转换：小范围到大范围，如 `int -> long`、`float -> double`。
2. 强制类型转换：大范围到小范围，如 `long -> int`、`double -> int`，可能丢失精度。
3. 字符串与数值的转换：如 `Integer.parseInt()`、`Double.parseDouble()`。
4. 包装类型与基本类型之间还有自动装箱与拆箱。

### 优秀回答范例

> Java 的类型转换主要有四类：第一类是隐式转换，比如 `int` 到 `long`；第二类是显式强转，比如 `double` 转 `int`，这时候要注意精度丢失；第三类是字符串和数值之间的转换，比如 `Integer.parseInt()`；第四类是包装类型和基本类型之间的自动装箱拆箱，比如 `int` 和 `Integer` 的转换。

### 常见追问

- `short s = 1; s = s + 1;` 为什么会报错？
- 自动拆箱可能引发什么空指针问题？

### 评分要点

- 是否覆盖隐式、显式、字符串转换、装箱拆箱。
- 是否能提到精度和溢出风险。

### 易错点

- 漏掉装箱拆箱。
- 只列 API，不讲转换规则。

## 条目 06：为什么金融计算常用 BigDecimal，不用 double

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：数值精度
- 关键词：BigDecimal、double、精度丢失、二进制浮点
- 难度：1

### 标准答案要点

1. `double` 使用二进制浮点表示，很多十进制小数无法精确表示，例如 `0.1`。
2. 金额计算对精度要求高，浮点误差会在累加、比较、结算场景中放大。
3. `BigDecimal` 适合高精度十进制运算，尤其适用于金额、税率、汇率等场景。
4. 使用 `BigDecimal` 时建议通过字符串构造，避免 `new BigDecimal(0.1)` 这类问题。

### 优秀回答范例

> `double` 的问题不是位数少，而是底层采用二进制浮点表示，很多十进制小数像 `0.1` 根本不能被精确表示，所以在金额计算里会出现误差。`BigDecimal` 是高精度十进制运算，更适合金融场景。实际开发里我会优先用字符串构造 `BigDecimal`，并配合统一舍入规则来保证结果可控。

### 常见追问

- `new BigDecimal("0.1")` 和 `new BigDecimal(0.1)` 有什么区别？
- `BigDecimal` 如何比较大小，为什么不推荐直接用 `equals()`？

### 评分要点

- 是否能讲清“二进制浮点”导致的误差根源。
- 是否知道 BigDecimal 的正确使用姿势。

### 易错点

- 只会说“double 不准”，但说不出为什么。
- 不了解 `compareTo()` 和 `equals()` 的差异。

## 条目 07：为什么 Java 要有 Integer

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：包装类型
- 关键词：Integer、int、包装类、泛型、集合、装箱拆箱
- 难度：1

### 标准答案要点

1. `int` 是基本类型，不是对象；`Integer` 是其包装类。
2. Java 很多 API、集合和泛型只能处理对象，不能直接使用基本类型。
3. 包装类还提供了工具方法，如 `parseInt()`、`valueOf()`。
4. 自动装箱与拆箱提高了开发便利性。

### 优秀回答范例

> Java 设计 `Integer` 的核心原因是面向对象体系很多地方只接受对象，比如集合和泛型不能直接放 `int`，所以需要一个包装类型。另外包装类本身也提供了很多工具方法，比如解析、比较和缓存机制。现在因为有自动装箱拆箱，开发时体验会更自然，但底层仍然区分基本类型和包装类型。

### 常见追问

- `Integer` 和 `int` 在内存和性能上有什么差异？
- 自动拆箱为什么会导致空指针？

### 评分要点

- 是否能结合集合、泛型和工具方法来说明价值。
- 是否理解本质上仍是两种不同类型。

### 易错点

- 误以为 `Integer` 只是“把 int 换个名字”。
- 不了解自动装箱拆箱带来的性能和空指针问题。

## 条目 08：说一下 Integer 的缓存机制

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：包装类机制
- 关键词：IntegerCache、valueOf、-128、127
- 难度：2

### 标准答案要点

1. `Integer` 内部有缓存池，默认缓存范围是 `-128 ~ 127`。
2. 通过 `Integer.valueOf()` 创建对象时，会优先从缓存中取。
3. 超出范围通常会创建新对象。
4. 这也是为什么某些场景下 `Integer` 用 `==` 比较会出现“有时 true、有时 false”。

### 优秀回答范例

> `Integer` 有一个默认缓存区间 `-128` 到 `127`。当我们调用 `Integer.valueOf()` 时，如果值在这个范围内，就会直接复用已有对象，而不是新建，所以这段区间内 `==` 可能返回 `true`。但我实际开发里不会依赖这个行为，包装类比较应优先使用 `equals()`，否则很容易踩坑。

### 常见追问

- `new Integer(127)` 和 `Integer.valueOf(127)` 有什么区别？
- 为什么包装类比较不建议用 `==`？

### 评分要点

- 是否准确说出缓存范围和 `valueOf()`。
- 是否能把缓存机制和 `==` 比较坑点联系起来。

### 易错点

- 认为所有 `Integer` 都会缓存。
- 不区分 `new` 和 `valueOf()`。

## 条目 09：抽象类和普通类有什么区别

- 岗位：Java后端开发
- 一级考点：面向对象
- 二级考点：抽象类
- 关键词：抽象类、实例化、模板方法、继承
- 难度：1

### 标准答案要点

1. 普通类可以直接实例化；抽象类不能直接实例化。
2. 抽象类可以包含抽象方法，也可以有具体方法和成员变量。
3. 抽象类常用来抽取公共逻辑，约束子类实现。
4. 普通类更偏具体实现，抽象类更偏模板与规范。

### 优秀回答范例

> 普通类可以直接 new 出对象，而抽象类不能，它更适合作为父类承载共性逻辑。抽象类既可以定义抽象方法强制子类实现，也可以提供默认实现、成员变量甚至构造方法，所以它比接口更适合表达“is-a”关系下的一组共同行为。

### 常见追问

- 抽象类可以有构造方法吗？
- 抽象类里能不能没有抽象方法？

### 评分要点

- 是否说清实例化差异和使用场景。
- 是否理解抽象类既能抽象也能复用代码。

### 易错点

- 以为抽象类里必须有抽象方法。
- 把抽象类和接口的使用场景混为一谈。

## 条目 10：抽象类和接口的区别是什么

- 岗位：Java后端开发
- 一级考点：面向对象
- 二级考点：抽象类与接口设计
- 关键词：interface、abstract、多实现、默认方法
- 难度：2

### 标准答案要点

1. 抽象类表达的是“是什么”，接口更偏“具备什么能力”。
2. 一个类只能继承一个抽象类，但可以实现多个接口。
3. 抽象类可以有实例变量、构造方法和具体实现；接口主要定义行为规范。
4. Java 8 以后接口可以有 `default` 和 `static` 方法，但依然不适合承载复杂状态。

### 优秀回答范例

> 我通常从设计语义和技术限制两个层面回答。抽象类适合抽取同一类事物的共性，比如统一模板、公共字段和默认实现；接口更像能力声明，比如可序列化、可比较。技术上一个类只能继承一个抽象类，但能实现多个接口，所以接口更利于解耦，抽象类更利于代码复用。

### 常见追问

- 什么时候优先用接口，什么时候优先用抽象类？
- Java 8 以后接口有默认方法，那抽象类还有什么价值？

### 评分要点

- 是否从“语义 + 技术限制 + 场景”三个维度回答。
- 是否能提到多实现和代码复用差异。

### 易错点

- 只背“一个单继承，一个多实现”。
- 忽略接口默认方法后的设计取舍。

## 条目 11：深拷贝和浅拷贝的区别

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：对象复制
- 关键词：深拷贝、浅拷贝、引用类型、共享对象
- 难度：2

### 标准答案要点

1. 浅拷贝复制的是对象本身，以及字段值；引用类型字段只复制引用地址。
2. 深拷贝会递归复制引用对象，得到完全独立的新对象。
3. 浅拷贝开销小，但共享子对象会带来联动修改风险。
4. 深拷贝适合需要强隔离的场景，如 DTO 快照、配置副本等。

### 优秀回答范例

> 浅拷贝只拷贝第一层对象，内部引用对象还是共用的；深拷贝会把引用对象也复制一份，所以修改副本时不会影响原对象。两者本质区别不是“拷得多不多”，而是引用对象是否共享。实际开发里如果对象层级复杂，我会谨慎评估性能和一致性成本，不会默认全量深拷贝。

### 常见追问

- `Object.clone()` 默认是深拷贝还是浅拷贝？
- JSON 序列化再反序列化算不算深拷贝？

### 评分要点

- 是否能说清共享引用这一核心点。
- 是否能结合实际场景说明取舍。

### 易错点

- 把“所有字段都复制”误解为一定是深拷贝。
- 不知道默认 `clone()` 通常是浅拷贝。

## 条目 12：什么是泛型

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：泛型与类型安全
- 关键词：泛型、类型安全、类型擦除、编译期检查
- 难度：2

### 标准答案要点

1. 泛型允许类、接口、方法在定义时使用类型参数。
2. 主要价值是提高编译期类型安全，减少强制类型转换。
3. Java 泛型底层基于类型擦除实现，运行时不会保留完整泛型参数信息。
4. 典型场景包括集合、通用工具方法、通用返回结果封装等。

### 优秀回答范例

> 泛型可以理解成把“类型”也抽成参数，常见如 `List<String>`。它的核心价值是把很多原本运行期才暴露的问题前移到编译期，比如类型不匹配。Java 泛型底层其实是类型擦除，所以编译后不会像 C# 那样完整保留泛型信息，这也是很多反射和重载限制的根源。

### 常见追问

- 什么是类型擦除？
- 为什么不能直接 `new T()`？
- 为什么不能用基本类型作为泛型参数？

### 评分要点

- 是否提到编译期类型安全。
- 是否知道 Java 泛型底层是类型擦除。

### 易错点

- 误说“编译后保留完整类型信息”。
- 不理解泛型和运行时类型的边界。

## 条目 13：new 出来的对象什么时候会被回收

- 岗位：Java后端开发
- 一级考点：JVM
- 二级考点：垃圾回收
- 关键词：GC、可达性分析、GC Roots、finalize
- 难度：2

### 标准答案要点

1. 对象不会因为“创建了很久”就回收，而是当它不可达时才可能被 GC 回收。
2. HotSpot 主要基于可达性分析判断对象是否存活，而不是引用计数法。
3. 只要对象从 `GC Roots` 可达，就不会被回收。
4. 具体何时回收由垃圾回收器策略和内存压力决定，不保证立刻发生。
5. `finalize()` 已不推荐依赖。

### 优秀回答范例

## 条目 18：try-catch-finally 的执行流程

- 岗位：Java后端开发
- 一级考点：异常机制
- 二级考点：异常处理流程
- 关键词：try、catch、finally、return、异常传播
- 难度：2

### 标准答案要点

1. `try` 中代码顺序执行，出现异常后后续语句中断。
2. JVM 会按顺序匹配合适的 `catch`，匹配到就执行。
3. `finally` 一般都会执行，常用于资源释放。

5. `return` 与 `finally` 同时存在时，需特别注意返回值与覆盖行为。

### 优秀回答范例

> `try-catch-finally` 的核心流程是：先执行 `try`，如果中间没有异常，就跳过 `catch` 直接进 `finally`；如果出现异常，就从后续语句中断，去匹配合适的 `catch`，匹配成功后执行对应处理逻辑，最后通常还会执行 `finally`。如果没有匹配的 `catch`，异常会继续往上抛给调用方。实际开发里我会把关闭连接、释放锁、清理上下文这种逻辑放在 `finally` 或 try-with-resources 里。

- try-with-resources 为什么更推荐？

### 评分要点

- 是否能说明异常中断后流程如何跳转。
- 是否知道 `finally` 的一般作用和边界情况。

### 易错点

- 说成 `catch` 执行完就结束，不提 `finally`。
- 不知道 try-with-resources。

## 条目 20：`hashCode()` 和 `equals()` 有什么关系

- 岗位：Java后端开发
- 一级考点：Java基础
- 二级考点：对象相等性约定
- 关键词：hashCode、equals、HashMap、哈希冲突
- 难度：2

### 标准答案要点

1. 如果两个对象 `equals()` 为 `true`，那么它们的 `hashCode()` 必须相同。
2. 如果两个对象 `hashCode()` 相同，`equals()` 不一定为 `true`，因为可能发生哈希冲突。
3. 重写 `equals()` 时通常必须同时重写 `hashCode()`。
4. 这是为了保证对象在 `HashMap`、`HashSet` 等基于哈希的容器中行为正确。

### 优秀回答范例

> `equals()` 解决的是“逻辑上是否相等”，`hashCode()` 解决的是“散列到哪个桶”。两者有明确约定：如果两个对象逻辑相等，那么它们的哈希值必须相同；但哈希值相同不代表一定相等，因为可能冲突。实际开发中如果只重写 `equals()` 不重写 `hashCode()`，放进 `HashSet` 或当作 `HashMap` key 时就可能出现查找异常。

### 常见追问

- 为什么 `HashMap` 先比较 `hashCode()` 再比较 `equals()`？
- 使用 Lombok `@Data` 时这个问题是怎么处理的？

### 评分要点

- 是否能说清必要条件和非充分条件。
- 是否能结合哈希容器说明后果。

### 易错点

- 误说“hashCode 一样就一定 equals 一样”。
- 不知道不同时重写会有什么问题。

## 条目 26：如果让你自己设计序列化和反序列化方案，你会怎么做

- 岗位：Java后端开发
- 一级考点：分布式基础
- 二级考点：序列化协议设计
- 关键词：序列化、反序列化、Protobuf、JSON、安全性、兼容性
- 难度：3

### 标准答案要点

1. 先明确目标：性能优先、可读性优先、跨语言优先还是兼容性优先。
2. 如果是内部高性能 RPC，优先考虑 `Protobuf`、`Kryo` 等高效二进制协议。
3. 如果是开放接口或调试友好型场景，常用 `JSON`。
4. 设计时要考虑字段编号、版本兼容、向前向后兼容、默认值、压缩、异常处理和安全校验。
5. 不推荐 Java 原生序列化用于生产核心链路，原因包括性能、安全和跨语言能力不足。

### 优秀回答范例

> 如果让我自己设计，我会先看场景。对内部服务调用，我会更倾向 Protobuf 这种二进制协议，因为体积小、性能高、跨语言支持也好；如果是开放接口，我可能会选择 JSON，换取可读性和调试便利。协议设计上我会重点控制几个点：字段编号稳定、版本兼容、默认值策略、异常字段兜底、安全校验，以及是否需要压缩和签名。Java 原生序列化我一般不会用于生产核心链路，因为它不够安全、跨语言能力差，而且产物通常偏大。

### 常见追问

- Protobuf 为什么比 JSON 更高效？
- 如何做协议升级兼容？
- 为什么反序列化会有安全风险？

### 评分要点

- 是否从“场景驱动”而不是“工具背诵”出发。
- 是否同时覆盖性能、安全、兼容、跨语言几个核心维度。

### 易错点

- 只回答“我会用 Protobuf”，但说不出为什么。
- 忽略版本兼容和安全问题。
