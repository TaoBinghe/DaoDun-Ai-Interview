## 条目 1：v-show 与 v-if 的区别

- 岗位：Web前端工程师
- 一级考点：Vue.js
- 二级考点：指令与渲染
- 关键词：v-show、v-if、条件渲染、DOM操作、性能
- 难度：1

### 标准答案要点

1. 控制手段不同：`v-show` 通过 CSS `display: none` 隐藏元素，元素始终保留在 DOM 中；`v-if` 是真正的条件渲染，通过添加或移除 DOM 节点来控制显示隐藏。
2. 编译过程不同：`v-if` 切换时伴随局部编译/卸载的过程，会销毁和重建内部的事件监听和子组件；`v-show` 只是简单的 CSS 切换。
3. 触发时机不同：`v-show` 由 `false` 变为 `true` 时不会触发组件的生命周期钩子；`v-if` 切换时会触发相应的生命周期（`beforeCreate`/`created`/`beforeMount`/`mounted` 或 `beforeDestroy`/`destroyed`）。
4. 性能消耗：`v-if` 有更高的切换消耗（每次切换都要重建/销毁）；`v-show` 有更高的初始渲染消耗（无论条件如何，元素总是被渲染）。

### 优秀回答范例

> `v-show` 和 `v-if` 都是 Vue 中控制元素显示隐藏的指令，但实现原理和使用场景不同。`v-show` 通过 CSS 的 `display` 属性切换，元素始终存在，适合频繁切换的场景；`v-if` 则是真正的条件渲染，元素在条件为假时会被完全移除，切换开销较大，但初始渲染时条件为假则不渲染，适合运行时条件很少改变的场景。例如，一个需要频繁显示/隐藏的弹窗可以用 `v-show`，而根据用户权限展示的不同模块则更适合用 `v-if`。

### 常见追问

- 为什么 `v-if` 切换时组件生命周期会重新执行？
- 在同一个元素上同时使用 `v-if` 和 `v-show` 会怎样？
- 如何选择使用 `v-if` 还是 `v-show`？
- `v-for` 和 `v-if` 为什么不能同时使用？

### 评分要点

- 是否准确说出两者的实现机制（CSS 隐藏 vs DOM 移除）。
- 是否能结合生命周期说明差异。
- 能否给出合理的使用场景建议。

### 易错点

- 误认为 `v-show` 是通过 `visibility: hidden` 实现。
- 忽略 `v-if` 的惰性渲染特性。
- 混淆初始渲染开销与切换开销。

## 条目 2：Vue 生命周期的理解及 created 和 mounted 的区别

- 岗位：Web前端工程师
- 一级考点：Vue.js
- 二级考点：生命周期
- 关键词：生命周期、created、mounted、钩子函数、数据请求
- 难度：2

### 标准答案要点

1. 生命周期是指 Vue 实例从创建到销毁的整个过程，包括初始化、数据观测、模板编译、挂载 DOM、更新、卸载等阶段。
2. 主要生命周期钩子：`beforeCreate`、`created`、`beforeMount`、`mounted`、`beforeUpdate`、`updated`、`beforeDestroy`、`destroyed`，以及 `activated`/`deactivated`（keep-alive）、`errorCaptured` 等。
3. `created` 钩子在实例创建完成后被调用，此时已完成数据观测、属性和方法的运算，但尚未挂载 DOM，`$el` 属性不可见。
4. `mounted` 钩子在实例挂载到 DOM 后调用，此时可以访问到真实的 DOM 元素，也适合进行依赖 DOM 的操作。
5. 在 `created` 中请求数据：可以更早获取数据，减少页面加载时间，但无法操作 DOM；在 `mounted` 中请求数据：可以操作 DOM，但可能延迟数据显示。
6. 通常，如果数据请求不依赖 DOM，优先在 `created` 中进行，以便尽早获取数据并触发渲染；若需要操作 DOM 或依赖子组件挂载完成，则放在 `mounted` 中。

### 优秀回答范例

> Vue 生命周期分为创建、挂载、更新、销毁四个阶段。`created` 是实例创建完成后立即调用，此时已经可以访问 `data`、`methods` 等，但 `$el` 还未生成，不能操作 DOM。`mounted` 是模板渲染成真实 DOM 后调用，可以操作 DOM，也意味着子组件已经挂载完成。关于数据请求：如果请求不依赖 DOM，我通常会放在 `created` 中，这样数据能尽早返回并触发视图更新；如果确实需要操作 DOM 或获取元素宽高等信息，才放在 `mounted` 里。另外，服务端渲染（SSR）环境下只有 `created` 可用，所以通用逻辑也常写在 `created` 中。

### 常见追问

- `beforeCreate` 和 `created` 之间发生了什么？
- 在 `mounted` 中一定能获取到所有子组件的 DOM 吗？
- `keep-alive` 组件下生命周期有何变化？
- 为什么不能在 `created` 中操作 DOM？

### 评分要点

- 是否准确描述各生命周期阶段。
- 能否清晰说明 `created` 和 `mounted` 的差异。
- 是否结合数据请求场景给出合理选择。

### 易错点

- 误以为 `created` 时 DOM 已经存在。
- 认为所有请求都应在 `mounted` 中发起。
- 忽略 `mounted` 不能保证子组件全部挂载完成（需用 `$nextTick`）。

## 条目 3：Vue 中组件和插件有什么区别？

- 岗位：Web前端工程师
- 一级考点：Vue.js
- 二级考点：组件化与插件化
- 关键词：组件、插件、Vue.use、全局注册、功能扩展
- 难度：2

### 标准答案要点

1. 组件（Component）是用来构成页面 UI 的独立可复用模块，通常包含模板、脚本和样式，通过局部或全局注册使用。
2. 插件（Plugin）是用来为 Vue 添加全局功能的方式，可以扩展 Vue 本身，例如添加全局方法/属性、指令、过滤器、混入，或者通过 `Vue.prototype` 挂载实例方法。
3. 注册方式不同：组件通过 `Vue.component()` 或组件内 `components` 选项注册；插件必须通过 `Vue.use()` 安装，且插件通常暴露一个 `install` 方法。
4. 使用场景：组件用于构建视图层；插件用于集成第三方库（如 vue-router、vuex）、提供全局工具函数、自定义指令等。

### 优秀回答范例

> 组件和插件是 Vue 中两个不同的概念。组件是构成页面的基本单元，每个 `.vue` 文件就是一个组件，通过复用组件可以提高开发效率。插件则是用来增强 Vue 功能的机制，它可以添加全局资源、指令、过滤器，或者给 Vue 原型挂载方法，通常需要调用 `Vue.use()` 来安装。例如，vue-router 是一个插件，它注册了全局组件 `router-view` 和 `router-link`，并给 Vue 实例注入了 `$route` 和 `$router`。而组件只是插件能力的一部分，插件可以做更多框架层面的扩展。

### 常见追问

- 如何编写一个自定义插件？
- 插件和混入（mixin）有什么区别？
- 一个插件中可以注册多个全局组件吗？
- 为什么插件需要使用 `Vue.use()` 而不是直接调用？

### 评分要点

- 是否准确区分组件和插件的定位。
- 是否能说出插件的常见用途和注册方式。
- 是否理解插件可以包含组件。

### 易错点

- 把插件和组件混为一谈。
- 不知道插件必须通过 `Vue.use` 安装。
- 认为插件只能添加全局方法。

## 条目 4：Vue 双向数据绑定的原理是什么？

- 岗位：Web前端工程师
- 一级考点：Vue.js
- 二级考点：响应式原理
- 关键词：双向绑定、v-model、Object.defineProperty、Proxy、发布订阅
- 难度：3

### 标准答案要点

1. 双向数据绑定是指数据变化驱动视图更新，视图变化（如表单输入）也能更新数据。
2. Vue 2.x 通过 `Object.defineProperty` 对数据对象的属性进行劫持，结合发布-订阅模式实现；Vue 3.x 改用 `Proxy` 实现更全面的代理。
3. 核心三部分：Observer（监听器）、Dep（依赖收集器）、Watcher（观察者）、Compiler（编译器）。
   - Observer 递归遍历数据对象，使用 `Object.defineProperty` 将属性转为 getter/setter，并在 getter 中收集依赖，setter 中触发更新。
   - Compiler 解析模板指令，替换数据，并绑定更新函数，同时实例化 Watcher。
   - Watcher 作为桥梁，当数据变化时，Dep 通知 Watcher，Watcher 调用回调更新视图。
4. 对于表单元素，`v-model` 是语法糖，它结合了动态 `value` 绑定和 `input` 事件监听，实现视图到数据的更新。

### 优秀回答范例

> Vue 的双向绑定基于数据劫持和发布-订阅模式。在 Vue 2 中，通过 `Object.defineProperty` 遍历 data 中的所有属性，为每个属性添加 getter 和 setter，并在 getter 中收集依赖（Dep 和 Watcher），在 setter 中通知变化。同时，编译器 Compiler 解析模板中的指令（如 `v-model`、`{{}}`），为每个依赖创建 Watcher。当数据变化时，setter 触发 Dep 通知所有 Watcher 更新视图。对于 `v-model`，它本质上是一个语法糖：绑定 value 属性并监听 input 事件，实现视图到数据的反向更新。Vue 3 改用 Proxy，可以更高效地监听动态属性添加和删除，解决了 `Object.defineProperty` 的一些限制。

### 常见追问

- `Object.defineProperty` 有什么缺陷？Vue 3 的 Proxy 如何改进？
- Vue 中数组的变动是如何被监听的？
- 什么是依赖收集？为什么要依赖收集？
- 手动实现一个简单的双向绑定需要怎么做？

### 评分要点

- 是否理解数据劫持的核心机制。
- 能否清晰描述 Observer、Dep、Watcher 的关系。
- 是否知道 `v-model` 的本质。

### 易错点

- 只提 `Object.defineProperty` 不提及 Dep 和 Watcher。
- 误以为 `v-model` 是双向绑定的全部。
- 忽略 Vue 对数组方法的特殊处理。

## 条目 5：Vue 常用的修饰符有哪些？有什么应用场景？

- 岗位：Web前端工程师
- 一级考点：Vue.js
- 二级考点：指令修饰符
- 关键词：修饰符、v-model、事件修饰符、按键修饰符、v-bind
- 难度：2

### 标准答案要点

1. 表单修饰符：`lazy`（change 后同步）、`trim`（自动过滤首尾空格）、`number`（自动转为数值类型）。
2. 事件修饰符：`stop`（阻止冒泡）、`prevent`（阻止默认事件）、`capture`（事件捕获模式）、`self`（仅当 event.target 是自身时触发）、`once`（只触发一次）、`passive`（提升滚动性能）、`native`（监听组件根元素原生事件）。
3. 鼠标按键修饰符：`left`、`right`、`middle`。
4. 键盘修饰符：`enter`、`tab`、`delete`、`esc`、`space`、`up`/`down`/`left`/`right`，以及系统修饰键 `ctrl`、`alt`、`shift`、`meta`。
5. `v-bind` 修饰符：`prop`（强制绑定为 DOM 属性）、`camel`（将 kebab-case 转为 camelCase）、`sync`（实现父子组件双向绑定）。
6. 应用场景举例：`.stop` 用于阻止点击事件冒泡；`.prevent` 用于阻止表单提交刷新；`.once` 用于一次性事件；`.sync` 用于子组件更新父组件数据的简写。

### 优秀回答范例

> Vue 提供了丰富的修饰符来简化常见操作。比如表单修饰符：`.lazy` 可以将 `v-model` 的同步时机从 input 事件改为 change 事件，适合输入后延迟处理的场景；`.trim` 自动去除首尾空格，常用于用户名字段；`.number` 自动将输入转为数值类型，适合年龄、数量等输入。事件修饰符最常用：`.stop` 和 `.prevent` 分别替代 `event.stopPropagation()` 和 `event.preventDefault()`，让模板更简洁；`.once` 适合只执行一次的初始化操作；`.native` 用于在自定义组件上监听原生事件，比如点击一个封装的按钮组件。`.sync` 修饰符是实现父子组件双向绑定的便捷方式，本质上是一个语法糖，子组件触发 `update:propName` 事件即可更新父组件数据。

### 常见追问

- `.sync` 修饰符的实现原理是什么？
- `.passive` 修饰符主要用于什么场景？
- 如何自定义键盘修饰符的别名？
- 在组件上使用 `v-model` 时，如何自定义 prop 和事件？

### 评分要点

- 是否覆盖主要类别的修饰符。
- 是否能结合具体场景说明用途。
- 是否理解 `.sync` 和 `.native` 等较复杂修饰符的用法。

### 易错点

- 混淆 `.stop` 和 `.prevent`。
- 忘记 `.native` 只能在组件上使用。
- 不知道 `.sync` 要求子组件触发 `update:prop` 事件。

## 条目 6：什么是虚拟 DOM？如何实现一个虚拟 DOM？

- 岗位：Web前端工程师
- 一级考点：Vue.js
- 二级考点：虚拟 DOM 与 diff 算法
- 关键词：虚拟 DOM、VNode、diff、性能优化、渲染
- 难度：3

### 标准答案要点

1. 虚拟 DOM（Virtual DOM）是用 JavaScript 对象来描述真实 DOM 结构的一种抽象，通常包含标签名、属性、子节点等信息。
2. 为什么需要虚拟 DOM：直接操作真实 DOM 代价高昂（浏览器重排重绘），虚拟 DOM 可以批量对比差异（diff），最小化 DOM 操作，提升性能；同时跨平台能力强（如渲染到 native）。
3. 实现一个简单的虚拟 DOM 需要：
   - 定义 VNode 类/对象，包含 `tag`、`props`、`children` 等属性。
   - 实现渲染函数（h 函数）将真实 DOM 转换成 VNode。
   - 实现 patch 函数，对比新旧 VNode 并更新真实 DOM（diff 算法）。
4. diff 算法的核心是同级比较，通过 key 优化列表更新，减少不必要的 DOM 操作。

### 优秀回答范例

> 虚拟 DOM 是一个轻量级的 JavaScript 对象，它对应着真实 DOM 的树形结构。操作虚拟 DOM 比操作真实 DOM 更快吗？不一定，但通过 diff 算法可以批量、最小化地更新真实 DOM，避免频繁的重排重绘。实现一个简易虚拟 DOM 包括：首先定义 VNode 结构，比如 `{ tag, props, children }`；然后实现 `createElement` 函数（即 h 函数）来生成 VNode；最后实现 `render` 函数将 VNode 渲染为真实 DOM，以及 `patch` 函数比较新旧 VNode 并更新。在实际框架中，diff 算法会做更多优化，如利用 key 复用节点、跳过静态节点等。Vue 2 使用的是 Snabbdom 库的 diff 算法，Vue 3 则重写了算法，并引入静态提升等优化。

### 常见追问

- 虚拟 DOM 一定比直接操作 DOM 快吗？为什么？
- diff 算法的原理是什么？时间复杂度是多少？
- Vue 中 key 的作用是什么？为什么不能用索引作为 key？
- 如何实现一个简单的 diff 算法？

### 评分要点

- 是否理解虚拟 DOM 的本质和目的。
- 能否描述虚拟 DOM 的基本结构和主要函数。
- 是否了解 diff 的核心思想（同级比较、key 优化）。

### 易错点

- 误以为虚拟 DOM 在任何场景下都比原生 DOM 快。
- 不知道 diff 算法只做同级比较。
- 忽略 key 在列表 diff 中的重要性。

## 条目 7：Vue 项目中如何解决跨域？

- 岗位：Web前端工程师
- 一级考点：工程化与网络
- 二级考点：跨域解决方案
- 关键词：跨域、CORS、proxy、JSONP、同源策略
- 难度：2

### 标准答案要点

1. 跨域是由浏览器的同源策略引起的，只有当协议、域名、端口完全相同时才允许请求。
2. 常见解决方案：
   - **CORS（跨域资源共享）**：后端在响应头中添加 `Access-Control-Allow-Origin` 等字段，允许指定来源访问。这是最标准的解决方案。
   - **JSONP**：利用 `<script>` 标签没有跨域限制，通过动态创建 script 标签加载数据，仅支持 GET 请求。
   - **Proxy 代理**：在开发环境下通过 webpack-dev-server 配置代理（`devServer.proxy`），将请求转发到目标服务器；生产环境可通过 Nginx 反向代理或后端接口转发。
3. Vue CLI 项目中配置代理：在 `vue.config.js` 中设置 `devServer.proxy`，将特定路径的请求代理到目标地址。
4. 生产环境跨域：通常由后端配置 CORS，或由 Nginx 统一处理跨域。

### 优秀回答范例

> 跨域是浏览器为了安全而施加的限制。在 Vue 项目开发中，最常用的方法是配置代理。比如在 `vue.config.js` 中设置 `devServer.proxy`，将 `/api` 的请求代理到后端服务器地址，这样浏览器访问的是同源地址，由开发服务器转发请求，从而绕过跨域。生产环境下，我会根据情况选择：如果后端支持 CORS，直接配置 `Access-Control-Allow-Origin`；如果不支持，可以通过 Nginx 反向代理，将前端和后端放在同一域下，或者由后端做一次转发。JSONP 现在用得较少，因为它只支持 GET 且有安全隐患。总之，跨域解决的核心是让请求符合同源策略或由服务端明确授权。

### 常见追问

- CORS 中简单请求和复杂请求的区别是什么？
- 代理配置如何解决跨域？原理是什么？
- 如何配置多个代理？
- 生产环境中如果前端和后端部署在不同域名，如何处理跨域？

### 评分要点

- 是否理解跨域产生的原因。
- 能否列举多种解决方案并说明适用场景。
- 是否掌握开发环境代理配置的方法。

### 易错点

- 认为跨域是后端问题，前端无法干预。
- 混淆开发环境和生产环境的解决方案。
- 误以为 JSONP 可以解决所有跨域请求。

## 条目 8：Vue 项目部署到服务器后刷新报 404 的原因及解决

- 岗位：Web前端工程师
- 一级考点：工程化与部署
- 二级考点：路由模式与服务器配置
- 关键词：404、history 模式、hash 模式、SPA、Nginx 配置
- 难度：2

### 标准答案要点

1. 问题现象：Vue 项目在本地运行正常，部署到服务器后，访问首页正常，但刷新非首页（如 `/login`）时出现 404。
2. 原因：Vue 是单页应用（SPA），所有路由由前端通过 JS 动态渲染，只有一个 `index.html` 入口。当使用 `history` 模式时，URL 路径是真实的路径（如 `/login`），刷新时浏览器会向服务器请求该路径的资源，而服务器上没有对应的文件，因此返回 404。
3. `hash` 模式不会出现此问题，因为 `#` 后的内容不会被发送到服务器，服务器始终返回 `index.html`。
4. 解决方案：配置服务器将所有前端路由重定向到 `index.html`。例如 Nginx 配置中添加 `try_files $uri $uri/ /index.html;`，Apache 配置 `FallbackResource` 等。
5. 注意：如果项目部署在子路径下，还需正确配置 `publicPath` 和 `base`。

### 优秀回答范例

> 部署后刷新 404 是 history 模式的经典问题。因为单页应用只有一个入口 `index.html`，路由变化是前端通过 history API 实现的，服务器并不知道这些路由。当用户直接访问 `/login` 时，服务器会去寻找 `/login` 这个资源，自然找不到。解决方法是让服务器在收到未知路径的请求时，都返回 `index.html`，把路由解析交给前端。比如 Nginx 中配置 `try_files $uri $uri/ /index.html;`。另外，如果使用 Node.js 作为后端，可以添加类似 `connect-history-api-fallback` 的中间件。hash 模式则天生规避了这个问题，但 URL 中带有 `#` 不太美观。实际项目中，我会根据 SEO 和美观需求选择模式，并确保服务器配置正确。
